# Travel Agent 后端架构：理想终态是什么样子

这份文档不写"哪里有bug"，写的是：**如果这套系统的每一层都按照代码里已经设计好的样子真正跑起来，它应该长成什么样。** 素材全部来自代码里已经写好的类、字段、方法——不是凭空想象，是把散落的设计意图串成一个完整故事。文末补一小节现状对照，但那不是重点。

---

## 一、这套系统想成为什么

`main.py` 顶部写着："基于 LangGraph + LLM 的智能旅行规划系统，提供可解释的行程推荐"。这句话里两个关键词决定了整个架构的设计取向：

**"LangGraph"** 意味着核心不是一个黑箱式的单体 LLM 调用，而是一张可以被拆解、可以被检视、可以在中途停下来的状态图——规划一次旅行不是一次 prompt 问答，是九个（以及后来加的路由分支）独立、职责单一的步骤依次接力。

**"可解释"** 意味着每一步为什么这么做、依据了什么数据、承担了什么风险，都应该能被追溯出来，而不是用户拿到一份行程却说不清楚它是怎么被生成的。

理想状态下，这套系统应该是：**一个可以被暂停、被审计、被追责、被量化评分的旅行规划引擎**，而不是一个"问完就忘、出错不知道错在哪"的聊天机器人套壳。

---

## 二、Agent 层：九个（+路由）单一职责的节点，串成一条可检视的流水线

`agent/` 目录下的模块划分本身就是这套理念的落地：

- **`graph.py`** 是流水线蓝图——每个节点只干一件事（收集偏好、标准化约束、搜索目的地、规划路线、查天气、估预算、合成行程、审安全、格式化输出），节点之间用条件边连接，而不是一个巨大的函数里 if-else 套 if-else。这样设计的好处是：任何一步出问题，能精确定位到是"约束标准化"错了还是"预算估算"错了，不需要在一整段线性代码里找断点。

- **`travel_agent.py`** 是这条流水线的"发车调度员"——它不关心每个节点内部怎么做，只负责把用户的输入送进流水线，把流水线跑起来产生的每一步进度，实时地转成用户能看懂的提示（"正在搜索目的地景点…"），流式吐给前端。理想情况下，这个调度员应该完全信任流水线的执行结果，不需要自己再维护一套并行的状态管理逻辑。

- **`tools.py`** 是流水线上真正干脏活累活的工具箱——地理编码、路线规划、天气查询、预算估算，每个工具都应该带着"这个数据来自哪里、有多大程度上可信"的标注（`uncertainty_flags`），这样最终生成的行程里，用户能看到"这条路线信息来自高德地图 MCP，数据较新"和"这个预算是基于历史规则估算，可能有偏差"的区分，而不是把所有信息一视同仁地端出来。

- **`prompts.py`** 是这套系统跟 LLM 对话的"话术脚本库"——每个需要 LLM 参与判断的环节（意图分类、偏好提取、约束标准化、行程问答）都有一份独立、可单独调优的 prompt，而不是把所有指令塞进一个巨大的系统提示词里。这样以后想单独提升"意图识别准确率"，只需要改一份 prompt，不会牵动其他环节。

- **`mcp_bridge.py`** 是跟高德地图对话的适配层——理想情况下，Agent 完全不需要知道自己在跟"高德的某个具体 REST 接口"打交道，它只需要说"我要查这个坐标附近的景点"，MCP 协议层负责把这句话翻译成高德懂的调用格式。这样如果以后想换成腾讯地图或者百度地图，理论上只需要换一个 MCP 服务端，Agent 侧的工具调用代码不需要改一行——**这才是用 MCP 而不是直接调 API 的真正价值**：不是为了炫技，是为了把"业务逻辑"和"具体供应商的接口细节"彻底解耦。

## 三、记忆系统：三层各管一段时间尺度，互不越界

`memory/` 目录设计的是一套教科书级的分层记忆架构，`MemoryManager`（`memory_manager.py`）是统一门面，底下管着四个互相配合的子系统：

**会话流水层**（`session_history.py`，落在 PostgreSQL）：负责"这次对话完整发生了什么"——原始的每一句用户输入、每一句 Agent 回复，按时间顺序落库。这一层的价值在于**可追溯**：用户切换到三天前的某次规划会话，应该能完整看到当时聊了什么、系统当时怎么回复的，就像翻聊天记录一样精确。

**工作记忆层**（`short_term.py`）：负责"当前这个任务还记得哪些细节"——目的地、天数、预算这些正在被填充的槽位信息，应该实时同步到这一层，供 Agent 在同一次规划过程中随时查阅，不需要每次都重新从对话历史里解析。这一层还应该管理 token 预算——当对话变长，旧的细节应该被自动压缩成摘要，而不是无限制地把全部历史堆进每次 LLM 调用的上下文里，否则成本和延迟都会失控。

**长期记忆层**（`long_term.py`）：负责"跨越多次会话，这个用户是什么样的人"——分成情节记忆（"上次推荐去杭州西湖，用户反馈很满意"）、知识记忆（"清明节杭州游客量大，建议避开西湖核心景区"）、用户画像（"这个用户偏好性价比住宿，讨厌红眼航班"）三类，用向量检索的方式，在新一轮规划开始时，主动把跟当前话题相关的历史片段捞出来，喂给 LLM 做参考。这一层是"可解释AI"里最容易被用户感知到价值的部分——如果系统能在新的一次规划里说"注意到您上次提到不吃辣，本次推荐已经避开重庆火锅这类重辣菜系"，用户会直接感受到"这个系统记得我"。

**上下文管理层**（`context_manager.py`）：负责"把上面三层揉合成一份 LLM 能消化的最终上下文"，按照 MemGPT 那套分层管理的思路，给系统提示词、工作记忆、检索结果、对话历史分别分配一个 token 预算比例（比如 20%/10%/30%/40%），任何一部分超出预算就主动裁剪，而不是让上下文无限膨胀直到触发 API 报错。

**Mem0 适配层**（`mem0_adapter.py`）：作为长期记忆的一个可插拔实现——理想情况下，这一层的存在意味着系统不绑定在自研的向量检索逻辑上，如果 Mem0 这类第三方记忆框架能提供更好的记忆整理能力（自动去重、自动总结、自动遗忘不重要的细节），系统应该能无痛切换过去，自研的 `long_term.py` 作为它不可用时的降级兜底。

这四层配合起来，理想状态是：**用户随口一提的偏好，系统会记住；用户上次问过的问题，系统能翻出来；当前正在填的表单，系统不会填一半就忘；不管对话多长，系统发给 LLM 的东西永远是"精炼过的重点"而不是"全部堆在一起"。**

## 四、安全审查层：模型能建议，但系统决定能不能做

`security/` 目录的设计原则写得很直白（`guard.py` 文件头）：**"模型可以建议，系统负责授权；工具结果是数据，不是指令；高风险动作必须可审计、可确认、可回滚。"** 这句话背后是一整套"零信任"式的授权体系：

**`guard.py`** 的 `SecurityGuard.authorize()` 应该是**每一次工具调用之前**都必须经过的关卡——先检查这个工具本身是否在白名单里（比如"删除文件"这种工具压根不该存在于旅行规划场景），再检查传进去的参数里有没有意外携带密钥、密码之类的敏感信息，再检查如果涉及访问外部网址，域名是否在可信列表里，最后判断这次调用是不是触碰了"预订""付款""支付"这类高风险动作——一旦命中，不是直接拒绝，而是标记成"需要人工确认"，交还给用户来点头。

**`permissions.py`** 负责定义"谁能做什么"的权限矩阵和敏感词库，理想情况下不同的用户角色（普通用户/管理员/被审核账号）应该能被赋予不同的操作范围，工具调用前先核对"这个角色允许触发这个风险等级的操作吗"。

**`human_in_loop.py`** 是把"高风险但并非绝对不能做"的操作，暂停下来交给人类决定的机制——生成一个带超时时间的确认请求，用户在界面上点"同意"或"拒绝"，系统据此继续或终止流程，而不是自己悄悄做了决定再告诉用户结果。

**`audit_logger.py`** 负责把每一次授权检查的结果（谁、什么时候、调用了什么、结果如何）落成审计日志——理想情况下这份日志应该是不可篡改、可导出、可用来事后复盘"这次规划为什么会推荐一个超出预算的方案"的完整证据链。

理想状态下，行程规划过程中如果 Agent 想调用"预订机票"这类工具（哪怕只是模拟），应该会被 `SecurityGuard` 拦下来，弹出确认框；如果某次工具调用的参数里意外携带了看起来像身份证号的字符串，应该被 Secret 检测规则挡下来，绝不会被静默地发送到外部 API。**"安全"不是事后补一个免责声明，是事前拦截。**

## 五、评估体系：让"这次规划好不好"变成一个可以量化、可以追踪趋势的数字

`evaluation/` 目录设计的是一整套多维度评分框架，理想状态下，这套系统运转起来应该长这样：

**推理过程评估**（`reasoning_eval.py`，DoVer 框架）：把一次完整的规划过程拆解成一段段可检验的推理步骤，检查每一步有没有走错——选错了工具、传错了参数、漏了该做的检查、多做了没必要的重复调用、执行顺序不对、凭空编造了不存在的信息（幻觉）、或者某一步耗时过长——每一类失败都应该被精确归因，而不是笼统地说"这次规划不好"。

**工具调用评估**（`tool_eval.py`，AgentWorld 框架）：专门检查"Agent 选对工具了吗、参数填对了吗、拿到的结果质量如何"——这是评估 Agent"手脚是否协调"的维度，跟"脑子想得对不对"（推理评估）是分开的两件事。

**RAG 检索评估**（`rag_eval.py`，FACT 框架）：专门检查长期记忆系统"捞回来的历史片段真的相关吗、生成的内容真的忠实于检索到的事实吗"——这是防止"记忆系统检索了不相关的东西，或者 LLM 编造了记忆里根本没有的内容"这类问题的防线。

**端到端评估**（`end_to_end.py`，RACE 框架）：站在用户视角评估最终产出的行程本身——这个行程是否真的可执行（景点之间的距离和时间是否现实）、成本估算是否合理、覆盖的兴趣点是否全面、整体是否好用。

**综合指标**（`comprehensive_metrics.py`）和**评估协调器**（`evaluation_runner.py`）把以上几类评分汇总成一份完整报告，理想情况下，这套评估应该能**持续运行在生产流量上**——每一次真实用户的规划请求结束后，自动跑一遍这些评估器，把分数记录下来，形成一条随时间变化的质量趋势线。当某个模型版本升级、某个 prompt 被调整之后，能立刻从这条趋势线上看出"这次改动是让系统变好了还是变差了"，而不是靠开发者主观感觉"好像还行"。

## 六、可观测性：把每一次规划变成一棵可以点开细看的树

`observability/langfuse_client.py` 设计的可观测性模型是这样一棵树：

```
Trace（一次完整会话，用 session_id 标识）
├── Span: preference_collector（这一步耗时多久、输入输出是什么）
├── Span: destination_search
│   ├── Event: 调用高德MCP搜索杭州附近景点
│   └── Event: 返回15个POI，去重后剩8个
├── Span: route_planner
├── ...
└── Score: 综合评估分数（连接第五节的评估体系）
```

理想状态下，开发者（甚至授权后的用户）应该能在 Langfuse 的界面上，点开任意一次历史规划，像看火焰图一样，看到"这次规划一共花了多少秒、卡在哪一步最久、每一步具体调用了什么工具、传了什么参数、拿到了什么结果、LLM 具体的输入输出原文是什么"。这才是"可解释AI"这个定位真正落地的样子——不是嘴上说"我们很透明"，是真的能把每一步摊开给人看。

`tools.py` 里设计的 `trace_tool_call` 装饰器，理想情况下应该包裹在**每一个**对外的工具函数外面——只要调用了 `search_places`、`estimate_route` 这类函数，不需要工具作者手写任何埋点代码，装饰器自动把这次调用记录成一个 Span，附带耗时、参数、返回结果、是否报错。这样可观测性是"自动获得"的，不是"额外维护"的负担。

## 七、把五层拼起来：一次理想的规划应该是什么体验

用户说"我想去杭州玩3天，预算3000"。理想情况下：

1. **入口**：`intent_router` 判断这是一次全新规划请求
2. **记忆介入**：系统先从长期记忆里检索"这个用户以前提过的偏好"（比如"不吃辣""喜欢历史文化"），悄悄补充进上下文
3. **偏好收集**：LLM 从这句话里提取出目的地、天数、预算，如果关键信息不全，`preference_collector` 触发追问，图在这一步暂停，状态被 checkpointer 存进数据库——哪怕用户十分钟后才回来继续，或者服务器在这期间重启过，对话都能从暂停的地方精确续上
4. **执行链路**：约束标准化 → 搜索目的地（每一次工具调用先过 `SecurityGuard` 检查，再被 Langfuse 记录成一个 Span）→ 路线规划 → 天气查询 → 预算估算，每一步的耗时、输入输出都被完整记录
5. **安全兜底**：如果某一步意外触发了"预订""付款"相关的操作意图，`human_in_loop` 机制弹出确认请求，等待用户点头才继续
6. **产出**：`itinerary_synthesizer` 生成最终行程，每条推荐都标注着数据来源和可信度
7. **记忆回写**：这次规划的关键信息（用户最终选定的目的地、预算是否够用）被写回长期记忆，供下次对话参考
8. **持续评分**：这次规划结束后，评估体系自动跑一遍推理质量、工具调用质量、端到端质量评分，汇总进这个系统的整体质量趋势
9. **审计留痕**：整个过程的每一个决策点、每一次工具调用、每一次安全检查结果，都在审计日志里留下不可篡改的记录

这套体系跑起来之后，**"可解释"** 不再是一句宣传语，而是体现在：任何一次规划出了问题，都能精确定位到是哪一层、哪一步、什么原因；任何一次用户投诉"为什么推荐了超预算的方案"，都能翻出完整的决策链路给出解释；任何一次系统升级，都能用评估体系的分数变化证明"这次升级到底是不是真的更好了"。

## 八、跟现状的差距（简要，非重点）

上面畅想的每一层，在代码里都**已经被设计和实现**——`SecurityGuard`、`Langfuse` 客户端、评估器基类和五个具体评估器、记忆系统的四个子层，全部是真实存在、逻辑完整的代码，不是纸上谈兵。差距只在于**"接线"**这一步：

- `TravelAgent.__init__` 接收 `safety_guard` 和 `langfuse_client` 两个参数，但整条 `plan_travel` 执行链路里，从未真正调用 `self.safety.authorize(...)` 或用 `self.langfuse` 记录任何 trace——这两个精心设计的子系统目前处于"被传进来但没人用"的状态。
- 图里真正生效的安全检查是 `safety_reviewer` 节点，它只做了一件更朴素的事——扫描最终生成的行程文本里有没有出现"预订/付款"之类的关键词，跟 `SecurityGuard` 设想的"工具调用前置授权"完全不是一个层级的防护。
- `evaluation/` 目录下的五个评估器彼此之间互相引用完整，但主服务代码（`main.py`/`travel_agent.py`/`graph.py`）里没有任何一处引用这个目录——它们是一套完整可用的离线评估工具，需要手动运行脚本，不会随生产流量自动打分。
- 记忆系统的四层本身都能独立工作，但今天的排查发现过 `get_enhanced_context` 里"先构建好完整上下文、再用 PG 查询结果直接覆盖掉其中一部分"的重复逻辑——说明多层之间的组合调用还有值得理清的地方。

把这些"已经造好但没接上电"的部件真正接进主流程，是这套系统从"设计完整"走到"理想终态"最后也最关键的一段路。
