# WanderMind 旅行代理项目技术文档

## 1. 项目定位

WanderMind 是一个面向旅行规划场景的 Agent 化应用。用户输入目的地、天数、预算和偏好后，系统会自动拆解需求、调用外部工具、组织行程，并把结果返回前端。

这个项目的重点不只是“生成一份行程”，而是把意图理解、上下文整理、工具调用、安全校验、记忆保存和评测反馈串成一个可持续迭代的闭环。

## 2. 总体架构

### 前端展示层

负责承接用户输入、展示规划结果、查看地图、时间线、会话记录和工具调用日志。

### 后端编排层

FastAPI 作为统一入口，把用户意图、会话状态、工具结果和安全检查串成一条可追踪的执行链路。

### 规划与审查层

系统采用“规划 / 审查双角色”的方式推进任务。规划角色负责决定下一步怎么走，审查角色负责检查输出是否安全、完整、可展示。

### 工具与数据层

通过 MCP / API 接入地图、路线、天气、攻略搜索等能力，并把原始结果和压缩结果分开保存。

## 3. 核心流程

1. 用户输入自然语言需求，例如“成都 3 天游玩，预算 3000，偏好美食和轻松节奏”。
2. 后端先做上下文整理：会话历史、当前约束、工具输出、用户偏好一起进入上下文管理器。
3. 规划角色根据当前状态决定下一步走哪个节点，必要时调用地图、路线、天气等能力。
4. 审查角色在最后做安全检查、格式整理和结果收口，避免把不稳定内容直接返回给前端。
5. 规划完成后，本轮结果、工具调用和记忆会写入持久化存储，方便后续追问和历史切换。

## 4. 角色与 Skill 设计

### 搜索 Skill

负责 POI 检索和地点补充，优先返回可直接用于规划的结果，而不是把原始大段数据直接塞给模型。

### 路线 Skill

负责根据地点顺序和交通方式做路线估算，输出更适合展示给用户的距离和耗时信息。

### 天气 Skill

负责把多天预报整理成简短描述，供行程安排和风险提示使用。

### 地图 Skill

负责地理定位和可视化数据准备，给前端地图展示和导航链路提供坐标信息。

## 5. 上下文工程

### 原始层

原始工具结果只落盘，不直接进入 Prompt，方便回看和排障。

### 压缩层

把 POI、路线、天气等数据压成结构化摘要，减少 token 浪费。

### 约束层

把预算、天数、节奏、偏好等硬软约束单独管理。

### 提示层

最终 Prompt 只放本轮任务真正需要的最小信息。

## 6. 记忆与会话

### 短期记忆

保存最近几轮对话和当前会话状态，保证追问时系统不会丢上下文。

### 语义记忆

保存稳定偏好和历史习惯，例如用户喜欢的节奏、餐饮倾向、出行风格等。

### 情景记忆

保存具体发生过的规划片段和成功案例，方便后续检索同类会话时复用经验。

### 过程记忆

保存常用规则、提示词模板和工具调用套路，属于方法层记忆。

## 7. 外部工具与地图

系统优先使用真实可用的数据源，而不是只靠静态 mock。

| 能力 | 作用 |
|---|---|
| 地点搜索 | 找到目的地周边可用 POI |
| 路线规划 | 估算地点之间的距离与耗时 |
| 天气查询 | 提供天气预报和风险提示 |
| 攻略搜索 | 补充更冷门地点的参考信息 |

前端地图页直接消费结构化坐标数据，用于展示标记、路线和导航信息。

## 8. 安全设计

### 输入侧

先识别缺失约束，信息不完整时先追问，不直接硬生成。

### 工具侧

做白名单、参数校验和结果校验，避免异常数据污染后续链路。

### 日志侧

对日志、记忆和 trace 做脱敏处理，避免敏感字段进入审计面板。

### 动作侧

把高风险动作拦在真正执行前，必要时触发确认流程。系统只提供建议和官方入口，不直接越过业务边界替用户完成危险动作。

## 9. 评测设计

### RACE

用来做场景化综合评分。参考动态权重思路，不同旅行场景会对计划完整度、可执行性、约束遵守和表达清晰度赋予不同权重。

### DoVer

用来做解释和归因检查。重点关注“为什么会得出这个结论”，以及中间理由是否前后一致。

### AgentWorld

用来检查行动顺序和状态流转。旅行规划不只看最终答案，还要看工具调用、节点跳转和回退路径是否合理。

### FACT

用来做事实和来源校验。正文里涉及的 POI、路线、天气、预算等信息都要能回到 source_map，避免凭空编造。

## 10. 设计取舍

### 为什么不继续增加角色数量

角色数量一多，调度复杂度会快速上升。当前方案把复杂度压在核心规划链路上，其他能力都沉到底层 Skill。

### 为什么要做上下文压缩

原始工具返回很长，如果直接进模型，很容易把窗口撑爆。压缩后既保留关键信息，也方便复用和排障。

### 为什么要保留会话历史

用户常常会继续追问、改预算、换节奏。没有会话历史，系统就只能单轮应答，很难形成真实可用的体验。

### 后续可以继续扩展什么

后面可以继续加强长短期记忆、检索排序、评测闭环和多城市行程编排，但前提是先把当前主链路跑稳。
