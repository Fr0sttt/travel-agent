# 06 - 安全防护与 HITL

> 这一份讲的是：Agent 不是“会做事就行”，还必须知道什么不能做、什么时候要停、什么时候要交给人确认。

## 1. 为什么安全是必须项

旅行 Agent 的输出会影响用户的真实决策：

- 预算
- 行程安排
- 酒店选择
- 交通方式

如果系统乱给建议，最少是体验差，严重时会误导用户。  
所以安全不是附加功能，而是上线门槛。

## 2. 安全要分几层

建议你这样讲：

1. 输入层安全
   - 检查是否有明显危险意图

2. 过程层安全
   - 监控工具调用是否合法

3. 输出层安全
   - 检查最终文本是否包含风险建议

4. 人工确认层
   - 对高风险动作必须 HITL

## 3. 输入层要防什么

主要防：

- 违法请求
- 明显不合理的旅行安排
- 伪造预订、付款、代操作

如果出现敏感请求，系统不能直接进入规划，而应该先拦截或提示。

## 4. 过程层要防什么

常见风险：

- 工具被乱调用
- 工具参数不合法
- 工具结果被错误解释
- 某一步失败后模型继续硬编

所以每个工具调用前后都要做校验：

1. 参数校验
2. 结果校验
3. 风险标记
4. 异常降级

## 5. 输出层要防什么

最终输出要检查：

- 是否出现危险承诺
- 是否出现代预订/代付款暗示
- 是否把不确定信息说成确定事实
- 是否漏掉风险提示

例如：

- “我已经帮你订好了” -> 不允许
- “门票一定有” -> 不允许
- “放心，不用确认” -> 不允许

## 6. HITL 怎么设计

HITL = Human In The Loop。  
可以理解为：系统觉得这件事有风险，就不要自己拍板，交给人确认。

适合 HITL 的场景：

- 涉及预订
- 涉及付款
- 涉及个人敏感信息
- 涉及高风险决策

### HITL 的状态设计

建议是：

1. `requires_confirmation = true`
2. 展示风险原因
3. 等用户确认
4. 用户确认后才继续

## 7. 怎么讲“安全和体验平衡”

面试官可能会担心：安全太多会不会影响体验？

你可以这么答：

> 安全不是把体验砍掉，而是把高风险动作拦在关键节点。低风险路径尽量流畅，高风险路径强制确认。这样既不会让系统太保守，也不会让系统乱跑。

## 8. 常见追问

### 追问 1：为什么不让模型自己判断安全？

因为模型会有幻觉，也会受上下文影响。安全应该尽量交给明确规则和状态机，而不是完全靠模型主观判断。

### 追问 2：怎么防止模型越权？

限制工具权限、限制节点权限、限制输出模板，并对关键操作加人工确认。

### 追问 3：如果输出不确定怎么办？

宁可给“未知”或“建议核实”，也不要编一个看起来很确定的答案。

## 9. 验证方式

1. 敏感请求是否被拦截
2. 高风险动作是否触发确认
3. 输出中是否保留风险提示
4. 工具越权是否能被发现

## 10. 一句话总结

> 安全防护的本质，是给 Agent 的能力加边界，让它能帮你做事，但不能替你做所有决定。
