# 04 - 多节点状态机与工具管控

> 这一份是整套系统最像“工程系统”的部分。你要讲的是：旅行规划任务不能靠一句 prompt 全包，必须靠状态机把步骤拆开，工具调用也要受控。

## 1. 为什么不是一个大模型直接输出

如果把整个旅行规划交给一次模型调用，会遇到这些问题：

1. 过程不可控
   - 不知道先搜什么、后搜什么

2. 状态不可恢复
   - 一旦中断，前面做过的事情丢了

3. 工具调用不可解释
   - 后面很难知道到底是哪个工具导致结果变差

4. 容易发散
   - 模型会一边想、一边编、一边补

所以更合理的是把流程拆成状态机。

## 2. 为什么是状态机而不是“纯 Agent 自由跑”

你面试时可以这样讲：

> 我们不是为了追新概念而上 ReAct 或 multi-agent，而是根据问题本身做约束。旅行规划有明确阶段：先理解意图，再收集约束，再找 POI，再算路线，再看天气，再估预算，最后合成行程。这种任务天然更适合固定节点流转，稳定性比完全自由的 agent loop 更高。

这句话非常重要，因为它体现的是“从问题出发”，不是“为了技术名词硬套”。

## 3. 推荐的节点拆分

可以拆成这几步：

1. `intent_router`
   - 判断新规划 / 澄清 / 问答 / 闲聊

2. `preference_collector`
   - 抽取用户偏好和缺失字段

3. `constraint_normalizer`
   - 统一约束结构

4. `destination_search`
   - 搜 POI

5. `route_planner`
   - 排路线

6. `weather_advisor`
   - 查询天气

7. `budget_estimator`
   - 算预算

8. `itinerary_synthesizer`
   - 合成行程

9. `safety_reviewer`
   - 做安全审查

10. `output_formatter`
   - 输出最终结果

## 4. 工具管控的核心原则

### 4.1 工具不是想调就调

每个节点只允许调用它职责范围内的工具。

例如：

- POI 节点只能搜景点
- 路线节点只能算路线
- 预算节点只能做估算

### 4.2 工具结果不直接裸奔进 prompt

要先做：

1. 结果校验
2. 字段压缩
3. 去噪
4. 归一化

### 4.3 工具失败要有降级

常见降级策略：

- POI 搜索失败 -> 回退到热门景点
- 路线失败 -> 用直线距离兜底
- 天气失败 -> 标注“未知”
- 预算失败 -> 给区间估算

## 5. 状态机怎么体现“可恢复”

状态机真正的价值是：

1. 节点执行结果可以持久化
2. 中断后可以从当前状态继续
3. 澄清轮次可以接着上次跑

你可以这样讲：

> 我们不是只存最后答案，而是存状态。状态机每过一个节点就更新一次状态，这样用户下次回来时，不需要从头开始，而是能接着当前节点继续走。

## 6. 工程上怎么控制复杂度

### 6.1 每个节点只做一件事

这是最重要的原则。

例如：

- `destination_search` 只负责找地点
- `route_planner` 只负责排顺序
- `budget_estimator` 只负责预算

### 6.2 每个节点都要有输出边界

节点输出必须固定字段，不能随手塞一堆临时变量。

### 6.3 每个节点都要能打日志

至少要能看出来：

- 进了哪个节点
- 出了什么结果
- 是否有降级
- 哪个工具失败了

## 7. 面试里的经典追问

### 追问 1：为什么不用多 agent 协作？

答法：

> 这个任务更像有明确工序的业务流程，不需要多个自治体互相辩论。多 agent 会增加协调成本和不确定性，而固定状态机更适合这种可分解、可验证的规划场景。

### 追问 2：如果想加 ReAct 怎么办？

答法：

> 可以在某个节点内部使用 ReAct 处理局部推理，比如在 POI 过滤或工具选择上做小步 reasoning，但整体流程还是建议保持状态机骨架。

### 追问 3：工具太多会不会乱？

答法：

> 不会，因为每个节点只暴露自己允许的工具，而且工具输出会先做统一包装再进入上下文。

## 8. 验证方式

1. 单节点是否可独立测试
2. 节点切换时状态是否保留
3. 工具失败时是否能降级
4. 中断后是否能续上

## 9. 一句话总结

> 状态机的价值，是把一个容易发散的旅行规划任务拆成一条可恢复、可观测、可降级的确定性执行链。
