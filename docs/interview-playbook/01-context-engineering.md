# 01 - 上下文工程与多约束收敛

> 这一份是面试里最好先讲清楚的点。你可以把它理解成：我们不是单纯“让模型回答”，而是先把旅行任务里的混乱输入整理成一个可执行、可验证、可压缩的上下文，再交给模型。

## 1. 这个点要解决什么

旅行规划不是普通聊天，它天然会出现四类冲突：

1. 信息太多
   - POI、路线、天气、预算一进来就会膨胀
   - 单轮可能就是几千 token

2. 约束太多
   - 天数、预算、出发地、目的地、偏好、同行人、节奏都会互相影响

3. 结果要可解释
   - 不能只给一个“看起来不错”的行程
   - 要能说清楚为什么选它、为什么没选别的

4. 冲突要能提前拦截
   - 比如预算太低、天数太少、目的地不明确
   - 不能让模型在错误前提下继续乱编

你在面试里可以直接这么说：

> 我们要解决的不是“让 LLM 想一想”，而是把旅行输入先做成一个可控的任务上下文，让系统在输入阶段就知道哪些是硬约束、哪些是软偏好、哪些必须追问，避免模型在错误前提下继续发散。

## 2. 总体思路

把上下文分成四层：

1. `RawContext`
   - 原始工具输出
   - 不直接喂给模型

2. `CompactContext`
   - 压缩后的结构化摘要
   - 节点之间传递

3. `ConstraintContext`
   - 硬约束、软偏好、冲突项、阻断条件

4. `PromptContext`
   - 最终进入模型的最小上下文

核心原则是：

- 原始数据可以大，但不能直接进 prompt
- 模型看到的必须是结构化、少字段、低歧义的内容
- 硬约束不满足时，直接阻断或追问

## 3. 面试时怎么讲成一条线

你可以按这条线讲：

1. 用户输入很散，比如“成都 3 天 3000 预算 想轻松一点”
2. 系统先做约束解析，抽出目的地、天数、预算、节奏
3. 再去拉 POI、路线、天气等工具结果
4. 工具结果先落到 RawContext，不直接拼给 LLM
5. 规则层把 RawContext 压成 CompactContext
6. ConstraintContext 专门管理冲突，比如预算超支、目的地缺失
7. 最后 PromptContext 只保留模型必须知道的最小信息

这样模型不会被几十个字段淹没，也不会把“数据很多”误以为“信息更聪明”。

## 4. 关键实现点

### 4.1 RawContext

适合存：

- 原始 POI 返回
- 原始路线 polyline
- 原始天气数据
- 原始预算明细

不适合直接给模型：

- 大 JSON
- 冗余字段
- 无关 metadata

### 4.2 CompactContext

压缩时只保留：

- 名称
- 坐标
- 分类
- 简短理由
- 来源

路线只保留：

- 起点
- 终点
- 距离
- 时长

天气只保留：

- 日期
- 温度
- 天气描述

### 4.3 ConstraintContext

建议分三类：

- 硬约束
  - 预算上限
  - 天数
  - 目的地

- 软偏好
  - 节奏
  - 同行人
  - 兴趣

- 冲突项
  - 预算超支
  - 信息缺失
  - POI 太少

### 4.4 PromptContext

最后进模型的上下文应该非常短，通常只保留：

- 用户目的
- 当前约束
- 已选 POI
- 已排路线摘要
- 天气风险
- 预算范围

## 5. 端到端流程

推荐你这样记：

1. `preference_collector`
   - 抽目的地、天数、预算、偏好

2. `constraint_normalizer`
   - 把自然语言转成统一约束结构

3. `destination_search`
   - 拉 POI，但先落 RawContext

4. `route_planner`
   - 计算路线，但只保留 compact route

5. `weather_advisor`
   - 只输出短天气摘要

6. `budget_estimator`
   - 输出预算区间和超支风险

7. `itinerary_synthesizer`
   - 只吃 PromptContext

## 6. 怎么验证

看四个指标就够了：

1. token 是否稳定
2. 结果里是否还保留无关字段
3. 冲突是否能提前追问
4. 大输入下是否还能保持输出质量

你可以这样回答面试官：

> 我会看压缩前后上下文长度、模型输入字段数、追问率、超支率，以及最终行程是否保持一致性。只要这四个指标稳定，就说明上下文工程是有效的。

## 7. 常见追问

### 追问 1：为什么不把所有工具结果直接喂给模型？

因为模型不是数据库。信息越多不一定越好，太多反而会让重点丢失，而且成本和不稳定性都会上来。

### 追问 2：为什么要做分层？

因为“原始数据”“压缩数据”“约束数据”“提示词数据”是四种完全不同的职责，混在一起以后很难维护，也很难调试。

### 追问 3：怎么处理预算超支？

先在 ConstraintContext 里标记冲突，再决定是追问、降级还是重算，不让模型直接硬凑。

## 8. 一句话总结

> 上下文工程的本质，不是把信息堆给模型，而是把任务先压缩成模型真正能理解、能执行、能自洽的最小问题。
