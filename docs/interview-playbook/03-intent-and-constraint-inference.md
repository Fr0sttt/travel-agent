# 03 - 意图识别与隐性约束推导

> 这一份讲的是：用户不会按表单说话，所以系统必须先识别意图，再把自然语言里的隐性约束补出来。

## 1. 这个点解决什么问题

用户输入通常不是结构化的，例如：

- “去成都玩三天，预算三千，别太赶”
- “上海周末随便逛逛，最好别排太满”
- “想带家人去杭州，吃的好一点”

这种输入里有三类信息：

1. 明说的
   - 地点、时间、预算

2. 暗说的
   - 节奏轻松
   - 更偏美食
   - 适合家庭

3. 没说但必须补的
   - 具体目的地
   - 出发日期
   - 同行人

系统如果不先做意图识别，就会把所有输入都当成“生成一份行程”，然后在缺失信息的情况下硬编。

## 2. 推荐的意图类型

面试里可以讲成四类：

1. 新规划
   - 用户想开一个新的旅行任务

2. 继续澄清
   - 用户在补充缺失信息

3. 行程问答
   - 用户在问当前方案里的细节

4. 闲聊
   - 用户不是在提新任务

这个分类很关键，因为它直接决定路由走向。

## 3. 隐性约束怎么推导

所谓隐性约束，就是用户没明说，但从表达里能推出来的东西。

例如：

- “别太赶” -> 节奏偏慢
- “带家人” -> 需要更稳妥，路线别太极限
- “预算三千” -> 住宿和门票要做约束
- “想吃好一点” -> 餐饮权重上升

可以把它们拆成：

1. 硬约束
   - 必须满足

2. 软约束
   - 尽量满足

3. 业务偏好
   - 对行程排序有影响，但不是硬门槛

## 4. 一个好的推导流程

### 第一步：抽槽

从自然语言里抽出：

- destination
- duration_days
- budget_cny
- companions
- pace_preference
- interests

### 第二步：补默认值

例如用户只说了“别太赶”，那就默认：

- pace_preference = moderate

### 第三步：检测缺失

如果缺少：

- 目的地
- 天数
- 预算

就应该追问，而不是强行规划。

### 第四步：判断意图

意图判断不是孤立分类，而是跟缺失字段一起看的。

例如：

- 有新的目的地 -> 新规划
- 在补刚才缺的字段 -> 继续澄清
- 在问“这家酒店为什么没选” -> 行程问答

## 5. 怎么讲“隐性约束推导”更像工程

你可以说：

> 我们不是只做关键词匹配，而是把自然语言先转成约束图。显式约束直接抽出来，隐性约束通过规则和模型补全，然后统一落到 ConstraintContext 里。这样后续路由、搜索、预算和安全检查都能读取同一份约束事实。

这比“我让模型自己理解”要强很多。

## 6. 典型例子

### 例子 1：别太赶

推导：

- pace_preference = moderate
- 每天景点数量减少
- 路线更注重顺路而不是最短

### 例子 2：带老人

推导：

- 交通偏保守
- 少爬坡、少换乘
- 多休息点

### 例子 3：预算三千

推导：

- 住宿和门票都要受限
- 不适合过多高价景点

## 7. 面试官常问什么

### 追问 1：为什么不用纯规则？

因为自然语言很灵活，纯规则覆盖不住“别太赶”“轻松一点”“别太折腾”这种表达。

### 追问 2：为什么不用纯模型？

纯模型容易漂，而且不稳定。最稳的是“模型抽槽 + 规则兜底 + 缺失追问”。

### 追问 3：如何避免误判？

一是给模型明确输出格式，二是对关键字段做校验，三是当置信度不够时宁可追问。

## 8. 验证方式

要看这几个结果：

1. 缺字段时是否会追问
2. 隐性偏好是否能稳定抽出来
3. 相同输入是否能得到一致的约束
4. 路由是否会因为误判而走错分支

## 9. 一句话总结

> 意图识别不是简单分类，而是把自然语言变成可执行约束的入口，先把“用户到底想干嘛”讲清楚，后面的系统才有资格开始干活。
