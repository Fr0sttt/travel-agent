# 05 - 多维度评测与自进化

> 这一份要讲清楚：一个 Agent 系统不是“跑通就完事”，而是要能评测、能发现问题、能迭代优化。

## 1. 为什么要做评测

Agent 系统的难点不是“有没有结果”，而是“结果好不好、稳不稳、可不可复现”。

如果没有评测，你就不知道：

- 哪一步出了问题
- 是工具错了，还是模型理解错了
- 是预算偏了，还是路线不合理
- 是个别 case 失败，还是系统性退化

所以评测不是附属品，而是系统的一部分。

## 2. 评测应该覆盖什么

建议分五类：

1. 意图识别评测
   - 是否抽对了目的地、预算、天数、偏好

2. 工具调用评测
   - 是否在正确节点调用了正确工具

3. 结果质量评测
   - 行程是否合理、可执行、连贯

4. 安全合规评测
   - 是否触发危险操作

5. 稳定性评测
   - 同样输入是否能得到一致结果

## 3. 一个好的评测流水线长什么样

### 第一步：准备测试集

测试集要覆盖：

- 不同城市
- 不同天数
- 不同预算
- 不同偏好
- 不同复杂度

### 第二步：定义 gold 标准

不是一定要绝对答案，而是定义“什么算好”。

例如：

- 预算不超上限
- 每天景点数量合理
- 路线不要来回折返
- 安全提示要出现

### 第三步：跑自动评测

每次改完系统，自动跑一轮：

- 成功率
- 追问率
- 工具命中率
- 预算偏差
- 风险触发率

### 第四步：输出分析报告

报告里要能看到：

- 哪些样例失败
- 失败发生在哪个节点
- 是模型问题还是工具问题
- 哪些规则需要更新

## 4. 自进化怎么理解

这里的“自进化”不是科幻词，而是：

1. 从失败样例里总结新规则
2. 把高频错误写成 prompt / rule / test
3. 下次再跑时验证是否修复

换句话说：

> 自进化不是让模型自己变聪明，而是让系统把失败经验沉淀成可执行的规则和测试。

## 5. 推荐的闭环

一个完整闭环可以这么讲：

1. 线上收集样例
2. 离线标注问题
3. 自动评测回放
4. 生成失败归因
5. 修 prompt / 修规则 / 修工具
6. 回归测试
7. 观察指标变化

这个闭环会让你在面试里显得非常“系统化”。

## 6. 常见指标

可以记这几个：

- Intent Accuracy
- Tool Success Rate
- Constraint Satisfaction Rate
- Itinerary Coherence Score
- Safety Trigger Coverage
- Regression Pass Rate

如果面试官想看你是不是懂评测，你就把这几个指标讲出来。

## 7. 怎么讲“多维度”

多维度不是“我看很多东西”，而是每个维度对应不同问题：

- 意图 -> 用户想干嘛
- 工具 -> 系统有没有做对
- 结果 -> 最终有没有用
- 安全 -> 有没有风险
- 稳定 -> 能不能复现

## 8. 面试追问

### 追问 1：为什么要做离线评测？

因为线上体验只能告诉你“坏了没有”，离线评测才能告诉你“为什么坏”。

### 追问 2：怎么避免评测集失真？

要定期更新样例，覆盖新的城市、新的玩法、新的失败模式。

### 追问 3：自进化和 prompt 调参有什么区别？

prompt 调参是局部优化，自进化是“发现问题 -> 固化规则 -> 形成回归测试”的闭环。

## 9. 验证方式

1. 新样例能不能覆盖老问题
2. 回归后失败率有没有下降
3. 指标有没有更稳定
4. 高风险 case 有没有被拦住

## 10. 一句话总结

> 评测的意义不是打分，而是把 Agent 系统变成一个能持续发现问题、归纳问题、修复问题的工程闭环。
