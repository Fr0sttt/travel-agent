# 02 - 记忆分层与跨会话延续

> 这一份要讲清楚：系统不是“每次重新问一遍”，而是能记住用户、记住会话、记住长期偏好，并且区分短期上下文和长期记忆。

## 1. 这个点要解决什么

旅行 Agent 的记忆如果不分层，会出现三个问题：

1. 会话越长越乱
   - 最近几轮对话和历史偏好混在一起

2. 记忆越多越慢
   - 所有历史消息都塞进 prompt，成本会越来越高

3. 记忆越久越不准
   - 很久以前的偏好未必还有效，不能无脑复用

所以记忆不是“存起来就完了”，而是要分层管理。

## 2. 推荐的记忆分层

可以按四层讲：

1. `ShortTermMemory`
   - 最近几轮会话
   - 当前上下文里最重要的信息

2. `SessionMemory`
   - 单个会话内的可恢复状态
   - 比如正在追问、当前目的地、临时补充条件

3. `LongTermMemory`
   - 跨会话稳定偏好
   - 比如“喜欢美食”“讨厌赶行程”“预算通常在 5000 左右”

4. `EpisodicMemory`
   - 历史旅行经历
   - 例如“上次去成都走了宽窄巷子，用户觉得太挤”

## 3. 面试时怎么讲

你可以这样说：

> 我们把记忆分成短期和长期两条线。短期记忆负责当前会话的连续性，长期记忆负责跨会话偏好和经验沉淀。每次对话不会把所有历史都拉进来，而是先做检索和过滤，再决定哪些记忆真正进入当前上下文。

这句话很重要，因为它把“记住”说成了一个工程流程，而不是一个模糊能力。

## 4. 完整落地流程

### 4.1 用户发起一次新请求

系统先做：

- 身份识别
- 会话识别
- 是否命中已有记忆

### 4.2 提取当前轮关键信息

从本轮消息里抽：

- 目的地
- 天数
- 预算
- 节奏
- 同行人
- 兴趣

### 4.3 判断哪些要写入记忆

不是所有东西都进长期记忆。建议区分：

- 稳定偏好：可以写
- 一次性临时条件：只进会话记忆
- 明显错误或犹豫信息：不写

### 4.4 记忆召回

在下一轮会话开始前：

1. 先查长期记忆
2. 再查最近会话状态
3. 再查本轮输入
4. 合并成最终上下文

这样能做到“上次说过喜欢轻松一点，这次不用重新问”。

## 5. 记忆写入策略

建议用“写前判断”：

1. 是否有明确偏好
2. 是否足够稳定
3. 是否可能过期
4. 是否值得记录

例如：

- “喜欢博物馆” -> 记
- “今天想随便逛逛” -> 只做会话级
- “这次只去成都” -> 只做会话级
- “预算通常 3000-5000” -> 可作为长期偏好

## 6. 记忆读取策略

不要直接把所有记忆读出来，建议：

1. 先按用户身份检索
2. 再按当前任务召回相关项
3. 再按相似度排序
4. 只保留 top-k

可用的判断标准：

- 跟当前目的地是否相关
- 跟当前预算是否接近
- 跟当前节奏是否冲突

## 7. 跨会话延续怎么讲

面试官常会问：

### “你怎么做到换个会话还能接着聊？”

你可以回答：

> 我会把会话状态持久化，不只是存最终结果，还存当前节点、澄清状态、最近几轮消息和关键约束。这样用户下次回来时，系统可以先恢复会话状态，再决定是继续追问、继续规划，还是开启新任务。

### “这和长期记忆有什么区别？”

回答：

> 会话状态解决的是“这一次任务跑到哪了”，长期记忆解决的是“这个人长期喜欢什么”。前者是连续性，后者是偏好沉淀。

## 8. 验证方式

你要盯这几个指标：

1. 用户第二次打开会话后是否还能看到上次状态
2. 是否能复用上次偏好
3. prompt 长度是否可控
4. 长期记忆是否误伤新的临时需求

## 9. 常见坑

### 坑 1：把所有消息都存进长期记忆

会导致记忆污染，越用越乱。

### 坑 2：长期记忆不做过期

用户去年喜欢的东西，今年不一定还喜欢。

### 坑 3：只存最终回复，不存过程状态

这样一旦中断，根本没法续上。

## 10. 一句话总结

> 记忆分层的核心不是“存更多”，而是“在正确的时间拿对的记忆，既能续上任务，又不污染当前决策”。
